Introduzione

Benvenuto nella guida operativa del nostro framework multi-agente stazionario. L'obiettivo primario di questo workflow è consentire la massima cooperazione tra agenti locali o globali, minimizzando l'inquinamento del contesto e prevenendo regressioni del codice dovuto alla saturazione dei token.

Il nostro framework implementa il modello di team multi-agente descritto da IBM nel video AI Agents Explained. IBM identifica 7 ruoli cognitivi fondamentali (Doer, Planner, Tool Operator, Learner, Critic, Supervisor, Presenter) che, lavorando in sinergia, permettono a un sistema AI di affrontare problemi complessi con la stessa precisione ed efficacia di un team umano specializzato.

Per garantire documentazioni aggiornate in tempo reale ed evitare allucinazioni su API deprecate, facciamo largo uso del Model Context Protocol (MCP). In particolare, integriamo strumenti come Google Stitch MCP per ispezionare i design system ed estrarre token di UI/UX per la creazione del file design.md (praticamente uno standard per lo sviluppo di interfacce generate da AI, come spiegato da Google in questo video: AI-Generated UI con Google Stitch).

Per il testing automatico e la validazione visuale del frontend, utilizziamo l'MCP di Playwright o WebMcp insieme a Dev Tools per Agenti, seguendo le linee guida ufficiali di Google descritte in Chrome DevTools for Agents e Modern Web Guidance (abilitando gli appositi flag sperimentali in Chrome e installando l'estensione, maggiori dettagli nel video Agent DevTools & Chrome).

2

Obiettivo

Costruire un team di 7 Sub-Agenti coordinati che replichi l'operatività di un team di sviluppo senior. Il fine ultimo è portare un'idea o feature da requisito verbale o formale a codice production-ready in autonomia e senza richiedere il monitoraggio o l'intervento continuo da parte dell'utente.

Qualità Enterprise

Scrittura di codice completamente testato, auto-documentato, privo di vulnerabilità e facilmente manutenibile a lungo termine.

Autonomia Orchestrata

Il Supervisor gestisce lo stato e delega i compiti in modo asincrono. L'utente umano interviene solo per milestone critici.

Specializzazione Massima

Ogni agente lavora unicamente nel proprio dominio logico, limitando allucinazioni, regressioni e conflitti di codice.

Tracciabilità Completa

Ogni singola decisione, modifica e progresso dello stato viene registrata in state.yaml e spiegata in linguaggio umano dal Presenter.

Struttura Iniziale

Nella radice del tuo progetto (che sia sviluppato in Vue, React, Laravel o qualsiasi altro stack), la struttura di configurazione iniziale dell'intelligenza artificiale varia in base allo strumento che decidi di utilizzare (Codex, Gemini, Antigravity, Claude Code, OpenCode, ecc.).

Non esiste una configurazione universale per tutti i client. Ad esempio, eseguendo il comando di inizializzazione (come /init), alcuni tool creano un file AGENTS.md nella radice del progetto per mappare le istruzioni stabili, mentre Claude Code genera un file CLAUDE.md con le regole di build e test. Stessa cosa per i server MCP, registrati solitamente in file JSON specifici di ciascun client, o per la definizione dei sub-agenti e delle loro skill (Codex adotta file .toml/.yaml e ragiona in termini di sub-agenti, mentre altri strumenti usano file .md).

Regola fondamentale: Date le notevoli discrepanze nei formati e nelle nomenclature tra i vari strumenti di Vibe Coding, si consiglia vivamente di leggere sempre la documentazione ufficiale del client specifico in uso.

Consiglio di portabilità: Se prevedi di riutilizzare gli stessi agenti a livello globale su tutti i tuoi progetti, evita di configurare le loro istruzioni localmente nella cartella di lavoro (es. /Users/NOME_USER/Sites/NOME_PROGETTO/.opencode/). Creali e posizionali invece nella cartella globale dell'utente (es. /Users/NOME_USER/.opencode/ o l'equivalente globale del tuo client).

Struttura delle Cartelle (Modello Logico)

Nota: La directory radice indicata come .ai/ rappresenta il contenitore logico che sul tuo progetto prenderà il nome specifico del tool (es. .gemini/, .opencode/, .claude/, .continue/, .aiassistant/, ecc.).

.ai/ — (può essere .gemini, .opencode, .claude, ecc.)
prd.md — Requisiti di Prodotto
architecture.md — Stack e Regole Tecniche
design.md — Token Visivi & UI Standards
workflow.md — Regole operative / Checkpoint
plan/
roadmap_YYYY-MM-DD.md — Roadmap attiva dei task
state.yaml — Puntatore dinamico volatile
.aiignore — Ignore specifico (es. .geminiignore)

Il file .aiignore varia a sua volta in base al client (ad esempio Gemini CLI adotta il file .geminiignore). Consulta sempre la documentazione per verificare il nome esatto. I restanti file di specifiche e piano sono costanti indispensabili per il corretto context management.

I file prd.md, architecture.md, e design.md fungono da riferimenti statici. Il file state.yaml (o state.md a seconda del formato preferito) è l'unico file volatile riscritto costantemente ad ogni checkpoint per spostare il puntatore logico del lavoro.

Nota sulla Gestione dei Plan:
Ciascuno strumento offre configurazioni specifiche per salvare le sessioni di piano nella cartella del progetto. Ad esempio:
  • Gemini CLI supporta directory e politiche di pianificazione personalizzate (vedi Custom Plan Directory).
  • Claude Code fornisce l'opzione plansDirectory per definire il percorso relativo di archiviazione dei file di pianificazione (es. ./plans, vedi Claude Settings).
YAML (.ai/plan/state.yaml) 
---
roadmap: ".ai/plan/roadmap_2026-06-17.md"
current_task: "T1.2"
next_agent: "developer-doer"
last_updated: "2026-06-17T20:30:00+02:00"
modified_files:
  - "src/components/ListItems.vue"
  - "src/stores/todo.js"
context_delta: "Autenticazione Mockata per completamento Frontend. Rilasciato con successo il modulo visuale."
---

I 7 Agenti Cognitivi (Modello IBM)

Abbiamo rielaborato i 7 ruoli teorici definiti da IBM in agenti operativi reali pronti all'uso. Ognuno di essi ha vincoli invalicabili, strumenti minimi ed è assegnato al modello di ragionamento più efficiente.

Capacità Design Architetturale
File in input Non definiti (Nuovi)
Modello Consigliato Claude Opus / GPT-4o / Gemini Pro

Istruzioni di Sistema (System Prompt):
Sei l'ARCHITECT-PLANNER del team. Il tuo obiettivo è analizzare la richiesta, valutare lo stato corrente e strutturare il design tecnico stazionando unicamente in .ai/prd.md, .ai/architecture.md e .ai/design.md. Non hai il permesso di scrivere codice o file applicativi al di fuori di .ai/. Utilizza le API MCP (es. Google Stitch) per estrarre regole di UI/UX, interfacce e stili prima di definire i contratti di sviluppo.

Capacità Slicing & Roadmap
File in input .ai/prd.md, architecture.md
Modello Consigliato Claude Sonnet / GPT-4o-mini

Istruzioni di Sistema (System Prompt):
Sei il TASK-MANAGER. Il tuo unico scopo è leggere i file strutturali in .ai/ e suddividerli in moduli logici sequenziali (Batch). All'interno di ciascun Batch, definisci task atomici (della durata stimata inferiore ai 30 minuti). Ogni task deve descrivere tassativamente: ID univoco, descrizione dell'output atteso, criteri di accettazione precisi e misurabili, "Context Files Pointer" (solo i file specifici che l'agente dovrà caricare in memoria) e il flag Parallel: [true/false].

Capacità Scrittura Codice & Test
File in input Puntati da Context Pointer
Modello Consigliato Claude Sonnet / Spark

Istruzioni di Sistema (System Prompt):
Sei il DEVELOPER-DOER. Il tuo compito è implementare il codice applicativo attenendoti al task corrente della roadmap. Sei isolato dal contesto totale: puoi leggere solo i file definiti nel "Context Files Pointer" e le linee guida stabili. Scrivi codice pulito, documentato e aggiungi test automatici per ogni logica introdotta. Al termine della scrittura, compila/linta localmente per validare. Non avviare deploy e non decidere modifiche di design.

Capacità Bash, Esecuzione comandi, CI/CD
File in input Script di build/test, config
Modello Consigliato Claude Haiku / GPT-4o-mini

Istruzioni di Sistema (System Prompt):
Sei il TOOL-OPERATOR. Gestisci le automazioni esterne, l'esecuzione delle suite di test, i linter, le build e le migrazioni di database. Non hai permessi per scrivere codice applicativo o business logic. Se un comando fallisce, restituisci l'output d'errore grezzo in modo strutturato (STATUS, OUTPUT, PROSSIMO STEP CONSIGLIATO) senza tentare refactoring.

Capacità Esplorazione & Web Search
File in input Tutto il workspace (Read-only)
Modello Consigliato Claude Sonnet / Gemini Flash

Istruzioni di Sistema (System Prompt):
Sei il LEARNER-RESEARCHER. Il tuo compito è navigare nella codebase per individuare pattern ricorrenti, interfacce API oscure, o consultare documentazioni di terze parti via Web Search o MCP (es. Context7). Genera esclusivamente memo testuali riassuntivi (<200 parole) contenenti link, sorgenti ed esempi di sintassi per supportare le attività del Developer o del Planner. Non puoi modificare file del workspace.

Capacità Code Review & Security
File in input Diff degli ultimi file modificati
Modello Consigliato Claude Opus / GPT-4o

Istruzioni di Sistema (System Prompt):
Sei il QA-CRITIC. Il tuo compito è impedire l'integrazione di codice non conforme o insicuro. Analizzi il codice modificato in base ai criteri di accettazione del task e a standard di sicurezza. Esegui verifiche su potenziali memory leak, query inefficienti e injection. Non scrivere codice. Assegna un punteggio da 1 a 10. Se il voto è inferiore a 8, rifiuta esplicitamente motivando le carenze.

Capacità Orchestratore di Flusso
File in input .ai/plan/state.yaml, roadmap.md
Modello Consigliato Claude Opus / GPT-4o

Istruzioni di Sistema (System Prompt):
Sei il SUPERVISOR-PRESENTER. Coordini l'intero ciclo di vita. Leggi lo stato iniziale, deleghi i compiti agli agenti specializzati, ricevi feedback dal QA-Critic e aggiorni .ai/plan/state.yaml richiamando il checkpoint. Alla chiusura del batch, assumi il ruolo di Presenter formulando un resoconto umano, chiaro e sintetico per l'utente, spiegando le modifiche architetturali e i file creati.

Il Workflow Operativo a 5 Fasi

La sequenza lineare garantisce che gli agenti mantengano stabilità cognitiva dall'analisi preliminare fino alla convalida.

1

Fase 1: Setup dell'Ambiente e Configurazione

Creazione delle directory fisiche del workspace e della cartella .ai/. Avvio del repository.

2

Fase 2: Sessione Blueprint (Progettazione)

L'agente architect-planner redige prd.md, architecture.md e design.md incorporando i token grafici da Stitch.

3

Fase 3: Pianificazione & Parallelismo

L'agente task-manager analizza la documentazione e genera la roadmap dei compiti con flags Parallel: true/false ed estrae state.yaml.

4

Fase 4: Esecuzione delle Sessioni Isolate

Sviluppo vero e proprio del codice. Il developer-doer legge solo i puntatori di file, esegue localmente, coordinato da supervisor-presenter.

5

Fase 5: Convalida (Quality Gate) e Checkpoint

Il qa-critic valida le modifiche. Se il verdetto è favorevole, il supervisor-presenter esegue il /checkpoint e azzera il contesto.

Integrazione Strumento: Global vs Local

Gli agenti intelligenti dovrebbero essere installati a livello globale sul tuo sistema, per evitare di duplicare configurazioni e system prompt ad ogni singolo repository. Le istruzioni sottostanti illustrano come procedere sia per il setup globale che locale.

Configurazione Globale (Consigliata)

Configura OpenCode a livello utente modificando il file di configurazione globale del client:

JSON (~/.config/opencode/opencode.json) 
{
  "globalAgentsPath": "~/.config/opencode/agents/AGENTS.md",
  "mcpServers": {
    "google-stitch": {
      "command": "npx",
      "args": ["-y", "@stitch/mcp-server", "--api-key", "STITCH_API_KEY"]
    },
    "context7": {
      "command": "npx",
      "args": ["-y", "@context7/mcp-server"]
    }
  }
}

Configurazione Locale

Se preferisci bloccare gli agenti sul singolo progetto, inizializza AGENTS.md nella root ed effettua il symlink dei contesti:

BASH 
# Crea symlink da .ai/ a .opencode/
ln -s .ai .opencode

# Genera file agenti locale
opencode /init

Configurazione Globale (Consigliata)

Posiziona i 7 file degli agenti in formato Markdown direttamente nella directory home di Claude:

BASH 
# Crea cartella agenti globale
mkdir -p ~/.claude/agents

# Copia i file degli agenti stabili nella directory globale
cp .ai/agents/*.md ~/.claude/agents/

Configurazione Locale

In Claude Code, le estensioni e gli agenti locali risiedono in .claude/agents/:

BASH 
mkdir -p .claude/agents
ln -s ../../.ai/agents/* .claude/agents/

Configurazione Globale (Consigliata)

Codex adotta file .yaml o .toml per descrivere i propri agenti ed impostare le risorse di calcolo.

BASH & Config TOML (~/.config/codex/agents/supervisor.toml) 
# Crea cartella agenti Codex globale
mkdir -p ~/.config/codex/agents/

# Esempio di file configurazione supervisor.toml globale:
name = "supervisor-presenter"
description = "Orchestra lo sviluppo leggendo state.yaml"
model = "gpt-4-turbo"
sandbox_mode = "isolated"
developer_instructions = """
Leggi .ai/plan/state.yaml. Spedisci istruzioni ai subagent.
Esegui checkpoint aggiornando i metadati YAML alla fine.
"""

La Gestione del Contesto v2 & Checkpoint

Il cuore operativo risiede nella rigida limitazione delle informazioni storiche per mantenere l'efficienza degli agenti.

Regola d'oro: Non alimentare l'AI con l'intero storico. Ogni sessione si apre, legge il file di stato, carica i file minimi dichiarati e si chiude al termine del singolo task azzerando la cache.

Il comando "continuiamo da dove abbiamo lasciato"

All'avvio di una nuova sessione isolata (con cache dei token svuotata a 0 tramite /clear o nuova chat), digita: 

> Tu: "continuiamo da dove abbiamo lasciato"

Cosa compie il Supervisor dietro le quinte:

  1. Accede a .ai/plan/state.yaml ed estrae l'indicazione della roadmap e del task attivo (es. T1.2).
  2. Verifica sulla roadmap i file contrassegnati sotto Context Files Pointer.
  3. Carica in memoria unicamente quei file ignorando il resto del repository.
  4. Spawna il sotto-agente descritto per quel task (es. developer-doer) ereditando le sue regole stabili.

Il protocollo /checkpoint

Quando un task di sviluppo verde viene approvato dall'agente qa-critic, si lancia il comando: 

> Tu: "/checkpoint"

Azioni eseguite dal Supervisor:

  1. Sovrascrive interamente .ai/plan/state.yaml con i nuovi dati stazionari (timestamp, ID del task completato, prossimo task e agente, lista dei file mutati).
  2. Il Supervisor assume il ruolo di Presenter riassumendo in linguaggio umano all'operatore cosa è stato fatto e il design risultante.
  3. Invita l'utente a digitare /clear per scaricare completamente la cache e iniziare il nuovo task in totale pulizia.
